AI × GitHub — ¿Qué te falta?

Revisión de integraciones IA disponibles en GitHub y servicios externos que potencian un portfolio de AI Engineer. Analizado sobre el proyecto py-agno-ai-workflow de @liezerfried.

Usando Parcial / Mejorable Falta completamente Opcional / Nice-to-have
🐙
GitHub — Features IA nativos
Herramientas que viven dentro de github.com y que cualquier reclutador puede ver directamente en tu repo
Contexto 2026: GitHub unificó Copilot, Models y Actions en un único ecosistema AI. El nuevo modelo de billing (desde junio 2026) cobra code reviews como AI Credits + minutos de Actions. Conocer esto diferencia a un AI Engineer de alguien que "usa Copilot".
GitHub Models
Falta
Marketplace con 100+ LLMs (GPT-4o, Llama 3, Mistral, Phi, Claude) accesibles gratis con tu GitHub PAT, usando el mismo SDK de OpenAI. Playground de comparación incluido.
Free tier OpenAI SDK compatible 100+ modelos
Cómo aplicarlo: Agregar GitHub Models como tercer provider en infrastructure/llm/provider.py. Muestra que podés benchmarkear modelos sin cambiar una línea del pipeline.
Copilot Code Review
Falta
Copilot revisa automáticamente cada PR que abrís en tu repo y posta comentarios de calidad. Desde junio 2026 consume AI Credits + Actions minutes — lo que significa que está integrado a nivel infraestructura, no es un plugin.
Desde jun 2026: billed Auto en cada PR
Por qué importa: Cualquier colaborador (o reclutador técnico) ve que tu repo tiene revisión IA activa — es señal de madurez de proceso.
Copilot Coding Agent
Opcional
Agente de Copilot que toma issues de GitHub, abre una rama, escribe código, y crea un PR. Incluye model picker, self-review, security scanning y CLI handoff. "GitHub Agentic Workflows" está en technical preview.
Technical preview Copilot Pro+
Oportunidad portfolio: Documentar en el README que tu proyecto tiene issues bien definidos para que el Coding Agent pueda procesarlos es un diferenciador.
GitHub Actions + AI
Falta
No hay ningún workflow .github/workflows/ en el repo. Cero CI/CD. El standard 2026 para un proyecto Python es: lint (ruff) + type check (mypy) + tests (pytest) corriendo automáticamente en cada push y PR.
Crítico para portfolio Free en repos públicos
Impacto inmediato: Un badge "CI passing" en el README es lo primero que mira un tech lead antes de leer una línea de código.
Dependabot + Security Alerts
Falta
GitHub puede escanear automáticamente tus dependencias (uv.lock) y abrirte PRs cuando aparecen CVEs. Gratis, cero configuración más allá de activarlo en Settings → Security.
Gratis Cero código
Cómo activar: Settings → Security → Dependabot alerts + Dependabot security updates → Enable.
GitHub Discussions / Wiki
Opcional
Tenés 16 archivos de docs pero están en /docs local. Moverlos a GitHub Wiki o activar Discussions permite que la conversación sobre decisiones de arquitectura sea pública y buscable — muestra pensamiento de product engineer.
Nice-to-have
Alternativa más rápida: Publicar el README con un link a cada doc clave en la sección "Architecture Decisions".
🤖
AI Code Review — Terceros
Apps que se instalan en tu GitHub y reviewan PRs automáticamente. Visibles en cada PR del repo.
CodeRabbit
Falta
La app de AI code review más instalada en GitHub (2M+ repos, 13M+ PRs procesados). Revisa cada PR con resumen en lenguaje natural, detecta bugs, sugiere mejoras, y soporta configuración via .coderabbit.yaml. Free tier para repos públicos.
Free para repos públicos 40+ linters integrados GitHub + GitLab + Bitbucket
Setup: Instalar desde GitHub Marketplace → agrega .coderabbit.yaml → cada PR recibe un review automático en minutos.
Qodo (ex CodiumAI)
Opcional
Alternativa a CodeRabbit. Genera tests automáticamente a partir de tu código, y revisa PRs. $30/user/mes en paid, pero tiene plan free. Fuerte en generación de tests — relevante dado que ya tenés un suite de pytest.
Test generation $30/user/mes (paid)
Recomendación: Empezar con CodeRabbit (gratis, más adoptado). Qodo vale si querés explorar generación de tests automáticos para el pipeline.
📊
LLM Observability
Trazabilidad de llamadas al LLM, latencia, tokens, evals. La diferencia entre "un script que llama a una API" y "un sistema de producción".
Estado actual: El proyecto usa Agno traces + openinference-instrumentation-agno. Eso es correcto y coherente con el stack. El gap es que no hay ninguna evidencia visible en el repo de que la observabilidad está activa y funcionando.
Agno Traces (app.agno.com)
Parcial
Ya tenés openinference-instrumentation-agno como dependencia y AGNO_API_KEY en el .env.example. Falta: un screenshot del dashboard en el README y documentar en el repo que el sistema está instrumentado.
Ya instalado No documentado en README
Quick win: Agregar una sección "Observability" en el README con un screenshot del trace de una corrección real. Proof > claims.
Langfuse
Alternativa
El estándar open-source de observabilidad LLM en 2026 (2300+ empresas, 19K stars). MIT license, self-hosteable. Cubre tracing, prompt versioning, evaluaciones LLM-as-judge, y métricas de producción. Integra con OpenTelemetry — compatible con tu stack.
MIT / Open source OpenTelemetry LLM-as-judge evals
Cuándo considerar: Si en el futuro querés mostrar métricas de evaluación (precision ≥ 0.85, hallucination_rate ≤ 0.05) con evidencia real trazada — Langfuse tiene el dashboard para eso.
Eval / Golden Dataset visible
Falta
Tenés tests/fixtures/golden_input.xlsx pero el resultado de la evaluación (precision, hallucination_rate) no se muestra en ningún lado. En 2026, un AI Engineer muestra métricas, no solo código.
Alto impacto portfolio Diferenciador AI Engineer
Implementación: Agregar un GitHub Actions job que corra el golden dataset y genere un badge o tabla de métricas en el README automáticamente.
⚙️
CI/CD — Lo mínimo que no podés no tener
No hay un solo archivo en .github/. Esto es el gap más grande y el más fácil de cerrar.
ci.yml — Lint + Type + Test
Falta
El workflow básico: correr ruff check, mypy y pytest en cada push y PR. Gratis en repos públicos. Sin esto, los 8 módulos de tests que escribiste no corren automáticamente en ningún lado.
Crítico ~30 líneas de YAML Free en repos públicos
Stack mínimo: uv sync → ruff check . → mypy agents/ infrastructure/ domain/ workflows/ → pytest tests/ -x
coverage.yml — Badge de cobertura
Falta
Agregar pytest-cov al pipeline y publicar el resultado como badge en el README. Un badge coverage: 87% en el repo dice más que cualquier descripción de "tengo tests de integración".
pytest-cov Codecov.io (free)
Setup: uv add --dev pytest-cov + registrar repo en codecov.io → badge automático.
pre-commit hooks
Falta
Tenés ruff configurado en pyproject.toml pero sin .pre-commit-config.yaml nadie lo corre antes de hacer commit. Garantiza que el código que llega al repo siempre pasa el linter.
ruff check ruff format mypy (si sumás type check)
Setup: Podés usar el skill /setup-pre-commit disponible en Claude Code para configurarlo automáticamente.
Dockerfile + docker-compose
Falta
El proyecto arranca con uv run chainlit run app.py. No hay historia de deployment. Para un proyecto de AI pipeline, Docker es el mínimo para demostrar que sabés llevar esto a producción real.
Medio impacto python:3.12-slim uv compatible
Mínimo: Dockerfile con python:3.12-slim + docker-compose.yml con las env vars documentadas.

Otros servicios que suman al portfolio
Fuera de GitHub, pero linkeable desde el README o perfil
app.agno.com — Dashboard público
Parcial
Agno tiene un dashboard de traces en app.agno.com que muestra las ejecuciones del workflow. Si activás el AGNO_API_KEY, podés linkearlo desde tu README como evidencia de que el sistema corre en producción.
Ya tienes la lib Solo falta el AGNO_API_KEY
Impacto: "Ver traces en vivo → app.agno.com/..." en el README es evidencia concreta de que es un sistema real, no un script.
HuggingFace Spaces — Demo pública
Opcional
HuggingFace Spaces permite hostear demos de AI gratis. Subir una versión simplificada del pipeline con un input de ejemplo hace que cualquier persona pueda probarlo sin clonar el repo — enorme diferenciador en portfolio.
Free tier Chainlit compatible Link directo en README
Alternativa más simple: Railway o Render — ambos tienen free tier y deploy directo desde GitHub.
LICENSE — MIT
Falta
El repo no tiene ninguna licencia. Sin licencia, técnicamente nadie puede reusar, forkar ni contribuir al código legalmente. 30 segundos para agregar un MIT.
Crítico si es open source GitHub lo genera automáticamente
Setup: GitHub → repo → Add file → Create new file → escribir LICENSE → GitHub sugiere templates → elegir MIT.
README — Badges + Demo GIF
Mejorable
El README tiene buena información arquitectónica pero le faltan badges de estado (CI, coverage, licencia) y un GIF o screenshot de la UI de Chainlit funcionando. El primer scroll de un visitante determina si sigue leyendo.
Alto impacto visual shields.io para badges
Mínimo a agregar: Badge CI · Badge Coverage · Badge License · 1 screenshot o GIF de la UI de Chainlit en acción.
